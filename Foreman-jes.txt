UPGRADE DEPENDENCIES
--------------------
1. Foreman Server must be upgraded BEFORE Smart Proxies
2. PostgreSQL 16 compatibility validated for all versions
3. Content must be synced before upgrade
4. All tasks must be completed before upgrade
5. Smart Proxies must be within N-2 version compatibility

SUPPORTED UPGRADE PATH
----------------------
Direct upgrade from 3.11 to 3.18 is NOT SUPPORTED.

Foreman follows a sequential upgrade model. Each minor version must be 
upgraded in sequence:

3.11 (Katello 4.13) -> 3.12 (Katello 4.14)
3.12 (Katello 4.14) -> 3.13 (Katello 4.14)
3.13 (Katello 4.14) -> 3.14 (Katello 4.14)
3.14 (Katello 4.14) -> 3.15 (Katello 4.15)
3.15 (Katello 4.15) -> 3.16 (Katello 4.15)
3.16 (Katello 4.15) -> 3.17 (Katello 4.15)
3.17 (Katello 4.15) -> 3.18 (Katello 4.15)

UPGRADE RATIONALE
-----------------
- Database schema changes require sequential migrations
- Pulp 3.x migrations are version-dependent
- Candlepin schema migrations are incremental
- foreman-maintain enforces version checks
- Skip upgrades risk data corruption

INTERMEDIATE VERSIONS REQUIRED
-------------------------------
Version   Katello   Release Date   Status
---------------------------------------------
3.11      4.13      Active         Source
3.12      4.14      Stable         Required
3.13      4.14      Stable         Required
3.14      4.14      Stable         Required

3.16      4.15      Stable         Required
3.17      4.15      Stable         Required
3.18      4.15      Stable         Target

ESTIMATED TIMELINE
------------------
Each hop: 30-60 minutes
Total hops: 7
Total estimated time: 4-7 hours
Recommended: 8-hour maintenance window

12.3 EMERGENCY ROLLBACK DECISION TREE
--------------------------------------------------------------------------------

START ROLLBACK IF:

CRITICAL - Immediate Rollback Required (< 15 minutes decision time):
✗ Database corruption detected
✗ Multiple services failing to start
✗ Data loss identified
✗ Security breach introduced by upgrade
✗ Complete UI unavailability > 30 minutes

✗ Host registration broken and unfixable
✗ Content synchronization completely broken
✗ Subscription management non-functional
✗ Critical provisioning workflows broken
✗ Performance degradation > 80%
✗ Pulp migration stuck > 4 hours

MEDIUM - Rollback Recommended (< 4 hours decision time):
✗ Non-critical features broken
✗ Performance degradation 50-80%
✗ Some Smart Proxies unreachable
✗ Partial functionality loss
✗ Workarounds not acceptable to business

LOW - Continue with Troubleshooting:
✗ Minor UI glitches
✗ Performance degradation < 50%
✗ Non-critical warnings in logs
✗ Acceptable workarounds available

ROLLBACK AUTHORIZATION PROCESS:

1. Technical Lead identifies rollback condition
2. Notify Change Manager immediately
3. Notify application stakeholders
4. Document specific failure condition
5. Obtain rollback authorization from Change Advisory Board (CAB)
6. Begin rollback procedure
7. Log all rollback actions
8. Post-rollback incident report required

SELECT blocked_locks.pid AS blocked_pid,
       blocked_activity.usename AS blocked_user,
       blocking_locks.pid AS blocking_pid,
       blocking_activity.usename AS blocking_user,
       blocked_activity.query AS blocked_statement,
       blocking_activity.query AS blocking_statement
FROM pg_catalog.pg_locks blocked_locks
JOIN pg_catalog.pg_stat_activity blocked_activity ON blocked_activity.pid = blocked_locks.pid
JOIN pg_catalog.pg_locks blocking_locks ON blocking_locks.locktype = blocked_locks.locktype
JOIN pg_catalog.pg_stat_activity blocking_activity ON blocking_activity.pid = blocking_locks.pid
