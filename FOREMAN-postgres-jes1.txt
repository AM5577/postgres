1.2 AUTO-UPGRADE STRATEGY EXPLAINED
--------------------------------------------------------------------------------

HOW FOREMAN-INSTALLER MANAGES POSTGRESQL:

1. Version Detection
   - Installer detects current PostgreSQL version
   - Determines if upgrade needed for target Foreman version
   - Plans upgrade path if required

2. Automatic Upgrade
   - Installer stops services
   - Backs up current database
   - Installs new PostgreSQL version
   - Runs pg_upgrade or equivalent migration
   - Updates configurations
   - Restarts services

3. Validation
   - Verifies database connectivity
   - Validates migrations
   - Confirms application functionality

WHEN POSTGRESQL AUTO-UPGRADE OCCURS:

The installer will upgrade PostgreSQL when:
- Target Foreman version requires newer PostgreSQL
- Current PostgreSQL version no longer supported
- Triggered automatically during foreman-installer execution

OPERATOR RESPONSIBILITIES:

□ Ensure sufficient disk space (2-3x database size)
□ Create backups before each hop
□ Monitor installer progress
□ Validate after each upgrade
□ Be prepared to rollback if issues occur

3.1 BACKUP PLANNING
--------------------------------------------------------------------------------

BACKUP COMPONENTS:

1. Foreman Application & Configuration
   - /etc/foreman
   - /etc/httpd (Foreman configs)
   - /etc/pulp
   - /etc/candlepin
   - /usr/share/foreman

2. PostgreSQL Databases
   - foreman database
   - pulpcore database
   - candlepin database
   - Database roles and globals

3. Pulp Content
   - /var/lib/pulp (may be very large)

4. System Configuration
   - Repository configurations
   - SSL certificates
   - SELinux contexts

BACKUP METHODS:

Primary: foreman-maintain backup (recommended)
Secondary: Manual database dumps
Tertiary: VM snapshots (if virtualized)

ESTIMATED BACKUP TIMES:

Small environment (< 500 hosts, < 100 GB content):
- foreman-maintain backup: 30-60 minutes
- Database dumps: 15-30 minutes
- VM snapshot: 5-15 minutes

Medium environment (500-5000 hosts, 100-500 GB content):
- foreman-maintain backup: 1-3 hours
- Database dumps: 30-90 minutes
- VM snapshot: 15-30 minutes

Large environment (> 5000 hosts, > 500 GB content):
- foreman-maintain backup: 3-8 hours
- Database dumps: 1-3 hours
- VM snapshot: 30-60 minutes

3.3 MANUAL DATABASE DUMPS (SECONDARY BACKUP)
--------------------------------------------------------------------------------

[FOREMAN SERVER]

# Create database backup directory
mkdir -p /var/foreman-upgrade/backups/database-dumps
chmod 700 /var/foreman-upgrade/backups/database-dumps

TIMESTAMP=$(date +%Y%m%d_%H%M%S)

# Dump foreman database
echo "Backing up foreman database..."

# Verify dump created
ls -lh /var/foreman-upgrade/backups/database-dumps/foreman_${TIMESTAMP}.dump

# Dump pulpcore database
echo "Backing up pulpcore database..."
sudo -u postgres pg_dump -Fc pulpcore > /var/foreman-upgrade/backups/database-dumps/pulpcore_${TIMESTAMP}.dump

# Verify dump created
ls -lh /var/foreman-upgrade/backups/database-dumps/pulpcore_${TIMESTAMP}.dump

# Dump candlepin database
echo "Backing up candlepin database..."
sudo -u postgres pg_dump -Fc candlepin > /var/foreman-upgrade/backups/database-dumps/candlepin_${TIMESTAMP}.dump

# Verify dump created
ls -lh /var/foreman-upgrade/backups/database-dumps/candlepin_${TIMESTAMP}.dump

# Backup database globals (roles, tablespaces, etc.)

# Check for bloat

sudo -u postgres psql -d foreman << 'EOF' > /var/foreman-upgrade/post-upgrade/table_bloat.txt
SELECT 
  schemaname,
  tablename,
  n_dead_tup,
  n_live_tup,
  ROUND(100.0 * n_dead_tup / NULLIF(n_live_tup + n_dead_tup, 0), 2) AS dead_tup_pct

WHERE n_dead_tup > 0
ORDER BY n_dead_tup DESC
LIMIT 20;
EOF

if [ "$BLOAT_COUNT" -gt 0 ]; then
  echo "Significant bloat detected. Running VACUUM..."
  sudo -u postgres vacuumdb -d foreman -v -z 2>&1 | tee /var/foreman-upgrade/post-upgrade/vacuum_foreman.log
  sudo -u postgres vacuumdb -d pulpcore -v -z 2>&1 | tee /var/foreman-upgrade/post-upgrade/vacuum_pulpcore.log
  sudo -u postgres vacuumdb -d candlepin -v -z 2>&1 | tee /var/foreman-upgrade/post-upgrade/vacuum_candlepin.log

15.1 SERVICE STARTUP FAILURES

========================================
ISSUE: Foreman Service Fails to Start
========================================

SYMPTOMS:
- systemctl status foreman shows "failed"
- Error in /var/log/foreman/production.log

DIAGNOSIS:

[FOREMAN SERVER]

systemctl status foreman -l --no-pager
journalctl -u foreman -n 100 --no-pager
tail -100 /var/log/foreman/production.log

ROOT CAUSES:
- Database connectivity failure
- Permission issues
- Port conflict
- SELinux denial

REMEDIATION:

# Test database connectivity
sudo -u foreman foreman-rake db:ping

# Fix permissions
sudo chown -R foreman:foreman /usr/share/foreman /var/log/foreman /var/lib/foreman
sudo restorecon -R /usr/share/foreman /var/log/foreman

# Check port availability
sudo ss -tulpn | grep 3000

# Check SELinux denials
sudo ausearch -m avc -ts recent | grep foreman

# If SELinux denials found
sudo audit2allow -a -M foreman_custom
sudo semodule -i foreman_custom.pp

# Restart service
sudo systemctl restart foreman

VALIDATION:
sudo systemctl is-active foreman
hammer ping

SYMPTOMS:
- systemctl status postgresql-XX shows "failed"
- Database connection errors

SELECT pid, relation::regclass, mode, granted

16.1 ROLLBACK DECISION CRITERIA
--------------------------------------------------------------------------------

EXECUTE ROLLBACK IF:

CRITICAL (Immediate Rollback):
✗ Database corruption detected
✗ Data loss > 5%
✗ All services failing to start
✗ Security breach introduced
✗ Complete UI unavailability > 1 hour

HIGH (Rollback Recommended):
✗ Critical functionality broken (host registration, content sync)
✗ Performance degradation > 80%
✗ Multiple services unstable
✗ Database connectivity failures

MEDIUM (Consider Rollback):
✗ Performance degradation 50-80%
✗ Non-critical features broken
✗ Workarounds not acceptable

LOW (Continue Troubleshooting):
✗ Minor UI issues
✗ Performance degradation < 50%
✗ Acceptable workarounds available
