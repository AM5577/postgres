UBUNTU LTS UPGRADE NOTES - PRODUCTION GUIDE
=============================================
Path: Ubuntu 22.04 -> 24.04 -> 26.04 (sequential only, no skipping)


=================================================================
QUICK START: END-TO-END WALKTHROUGH (EASY STEPS)
=================================================================
Follow these steps in order. Each step is simple - do not skip any.

STEP 1 - Check what version you're running
  lsb_release -a

STEP 2 - Make sure you have enough free disk space (need 10GB+)
  df -h

STEP 3 - Take a VMware snapshot of the VM (your safety net)
  -> See Section 5 for click-by-click steps.
  -> Name it: pre-upgrade-<current-version>-<date>

STEP 4 - Update the current system completely
  sudo apt update
  sudo apt upgrade -y
  sudo apt dist-upgrade -y
  sudo apt autoremove -y
  sudo reboot

STEP 5 - Install the upgrade tool
  sudo apt install update-manager-core -y

STEP 6 - Open a session that won't die if your connection drops
  screen -S upgrade
  (or: tmux new -s upgrade)

STEP 7 - Start the upgrade
  sudo do-release-upgrade
  -> Answer "yes" to prompts to continue
  -> When asked about config files, choose "keep local" unless
     you know you need the new default

STEP 8 - Reboot when the upgrade finishes
  sudo reboot

STEP 9 - Confirm the new version is correct
  lsb_release -a

STEP 10 - Run health checks (see Step 5 in Section 1 for full list)
  - dpkg -C            (no broken packages)
  - systemctl --failed (no failed services)
  - sudo apt-get update (no errors)

STEP 11 - If 24.04 is stable for 1-2 weeks, repeat Steps 1-10
  to go from 24.04 -> 26.04 (take a NEW snapshot first).

STEP 12 - If anything goes wrong
  -> Power off the VM and revert to the VMware snapshot
     from Step 3. Then investigate before retrying.

That's it - the sections below give full detail for each step.


=================================================================
SECTION 1: OFFICIAL PRODUCTION-GRADE IN-PLACE UPGRADE PROCEDURE
=================================================================

--- STEP 1: PRE-UPGRADE ASSESSMENT ---
1. Verify current version:
     lsb_release -a
     cat /etc/os-release

2. Check disk space (root needs >=10GB free, /boot >=200MB free):
     df -h
     df -i

3. Check hardware/arch:
     uname -m          # must be x86_64
     getconf LONG_BIT   # must be 64
     free -h

4. Detect virtualization (for snapshot planning):
     systemd-detect-virt

5. Review list of installed packages and held-back packages:
     apt list --upgradable
     dpkg -l | grep -v "^ii\|^rc"

--- STEP 2: BACKUP / SNAPSHOT (MANDATORY) ---
1. Take a VMware snapshot BEFORE proceeding (see Section 5).
2. Do NOT continue without a confirmed, restorable snapshot.

--- STEP 3: PRE-UPGRADE SYSTEM HEALTH CHECKS ---
1. Update current system fully first:
     sudo apt update
     sudo apt upgrade -y
     sudo apt dist-upgrade -y
     sudo apt autoremove -y

2. Reboot to apply pending kernel/updates:
     sudo reboot

3. Ensure update-manager-core is installed:
     sudo apt install update-manager-core -y

4. Check for held/broken packages:
     sudo dpkg -C
     sudo apt-get check

5. Stop non-critical services (apps, containers) to avoid
   conflicts during upgrade. Drain Kubernetes nodes if applicable.

--- STEP 4: RUN THE IN-PLACE UPGRADE ---
1. Confirm release upgrade tool is configured for LTS-to-LTS:
     cat /etc/update-manager/release-upgrades
     (Prompt= should be "lts")

2. Run the upgrade in a session that survives SSH drops.
   Use ONE of the following:

   Option A - screen:
     screen -S upgrade
     sudo do-release-upgrade
     (Detach: Ctrl+A then D | Reattach: screen -r upgrade)

   Option B - tmux:
     tmux new -s upgrade
     sudo do-release-upgrade
     (Detach: Ctrl+B then D | Reattach: tmux attach -t upgrade)

   Option C - nohup (no interactive prompts, non-interactive mode):
     sudo nohup do-release-upgrade -f DistUpgradeViewNonInteractive \
       > /var/log/do-release-upgrade-nohup.log 2>&1 &
     (Check progress: tail -f /var/log/do-release-upgrade-nohup.log)
     NOTE: Non-interactive mode auto-accepts default answers;
     review config-file conflicts manually after completion.

3. Follow prompts:
     - Accept repository changes
     - Keep or replace modified config files (review diffs)
     - Allow removal of obsolete packages when prompted

4. Upgrade completes and prompts for reboot:
     sudo reboot

--- STEP 5: POST-UPGRADE VALIDATION ---
1. Confirm new version:
     lsb_release -a

2. Run quick validation checks:
     - OS version matches target
     - Kernel version (uname -r)
     - No broken packages (dpkg -C)
     - No failed services (systemctl --failed)
     - Critical services: ssh, cron, rsyslog active
     - Disk space and network connectivity OK
     - sudo apt-get update completes with no errors

3. Verify application-specific items:
     - Rebuild Python virtualenvs (Python version changed)
     - Verify Docker/containers start correctly
     - Verify database services (PostgreSQL/MySQL) running;
       run pg_upgradecluster if needed
     - Update bootloader: sudo update-grub
     - Rebuild initramfs if needed:
         sudo update-initramfs -u -k all

4. Review upgrade logs:
     /var/log/dist-upgrade/main.log
     /var/log/dist-upgrade/apt.log

--- STEP 6: REPEAT FOR NEXT LTS HOP ---
1. Once 24.04 is stable for a soak period (1-2 weeks recommended
   in production), repeat Steps 1-5 for 24.04 -> 26.04.
2. Take a fresh, separate VMware snapshot before this hop.

--- STEP 7: ROLLBACK (IF UPGRADE FAILS) ---
1. Power off VM.
2. Revert to the pre-upgrade VMware snapshot
   (Snapshot Manager > Revert to Snapshot).
3. Power on and re-validate before retrying.

SECTION 3: UBUNTU 26.04 LTS (Resolute Raccoon) - RELEASED
=================================================================
- Released: 23 April 2026
- Support: 5 years standard (until April 2031); 10 years with
  Ubuntu Pro (ESM)
- Minimum requirements (Desktop): 2 GHz dual-core CPU, 6 GB RAM,
  25 GB free storage
- Minimum requirements (Server): from 1.5 GB RAM, 4 GB storage
  (scales with use case)
- Upgrade path: 24.04 LTS -> 26.04 LTS
  (22.04 must go through 24.04 first; cannot skip)
- Upgrade only after 24.04 is stable and validated.
- Upgrade process:
    sudo apt update && sudo apt upgrade -y
    sudo do-release-upgrade
- Run quick-validate.sh after upgrade to confirm health.
- Release notes: https://documentation.ubuntu.com/release-notes/26.04/

    sudo apt update

SECTION 5: BACKUP - VMWARE SNAPSHOTS
=================================================================
- Take a VM snapshot BEFORE every upgrade step
  (22.04->24.04 and 24.04->26.04).
- Steps (vSphere/Workstation):
    1. Power off or quiesce the VM (or use memory snapshot for
       running VM).
    2. Right-click VM > Snapshot > Take Snapshot.
    3. Name clearly, e.g.:
         "pre-upgrade-22.04-to-24.04-<date>"
         "pre-upgrade-24.04-to-26.04-<date>"
    4. Confirm snapshot completes and disk space is sufficient
       on datastore.
- Rollback: Right-click VM > Snapshot > Revert to Snapshot.
- VM Snapshot rollback reliability: EXCELLENT
  (works on VMware, Proxmox, KVM).
- Do NOT proceed with upgrade without a confirmed working
  snapshot.

--- 6.4 Syncing Repositories ---
1. Content --> Products --> [Product] --> Repositories
2. Select checkboxes for repos to sync (or select all).
3. Click "Sync Now".
4. Monitor: Content --> Sync Status
     - Green "Success" = OK
     - Red "Failed"    = check URL/network
5. Optional recurring sync:
     Content --> Sync Plans --> New Sync Plan --> assign to product
6. Verify content after sync:
     Content --> Products --> [Product] --> Repositories
     --> Click repo name --> "Packages" tab

--- 6.5 Removing / Deleting an Old Product (e.g., Ubuntu 20.04) ---
** Complete Pre-checks in 6.6 FIRST **

Step A: Remove all repositories from the product
  1. Content --> Products --> Ubuntu_20.04
  2. Repositories tab --> check ALL repos
     (focal, focal-backports, focal_pbis, focal-security,
      focal-updates)
  3. Click "Remove Repositories" --> Confirm.
  4. Monitor: Monitor --> Tasks (filter "Running"/"Error")

Step B: Delete the product
  1. Content --> Products
  2. Check box next to "Ubuntu_20.04"
  3. "Select Action" --> "Delete" --> Confirm
  4. Verify product no longer listed

--- 6.6 Pre-checks Before Deleting a Product ---
CHECK 1 - Content Views:
  Content --> Content Views
  - Ensure no Content View includes the old repos.
  - If found: edit, remove repos, publish new version, promote.

CHECK 2 - Activation Keys:
  Content --> Activation Keys
  - Ensure none reference old repos/Content Views.
  - If found: update/delete the activation key first.

CHECK 3 - Hosts:
  Hosts --> All Hosts
  - Check for hosts subscribed to the old product.
  - If found: migrate or unregister hosts first.

CHECK 4 - Sync Plans:
  Content --> Sync Plans
  - Remove old product from any assigned sync plans.

--- 6.7 Foreman Notes & Tips ---
- Always take a Foreman DB backup before mass deletions:
    foreman-maintain backup offline --skip-pulp-content /backup/
- "Reclaim Space" button in Repositories tab frees disk space
  after removing repos.
- Tasks: Monitor --> Tasks (filter "Running"/"Error")
- For large sync jobs, run during off-peak hours.
