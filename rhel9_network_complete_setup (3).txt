--------------------------------------------------------------------------------
11.6 TROUBLESHOOT: ROUTE NOT REMOVING - NM vs KERNEL MISMATCH
--------------------------------------------------------------------------------

PROBLEM:
  Route shows in "ip r" but won't remove with nmcli

DIAGNOSIS:
  # Check what NM actually has stored (may differ from ip r)
  nmcli connection show ens160 | grep route

  FINDING: NM had "10.10.10.0/21" stored but "ip r" showed "10.10.8.0/21"
           These are DIFFERENT routes - that's why removal failed!

FIX STEP 1: Remove the correct NM stored route
  nmcli connection modify ens160 -ipv4.routes "10.10.10.0/21 192.168.121.2"

FIX STEP 2: Full connection reset to clear kernel leftovers
  nmcli connection down ens160
  nmcli connection up ens160

FIX STEP 3: If route still in kernel after NM reset - remove manually
  ip route del 10.10.8.0/21

  NOTE: This is temporary. If NM does not have it, it will not come back on reboot.

# Verify both NM and kernel are clean
nmcli connection show ens160 | grep route
ip r

RESULT:
  default via 192.168.121.2 dev ens160 proto static metric 100
  1.1.0.0/21 via 192.168.121.2 dev ens160 proto static metric 100
  192.168.121.0/24 dev ens160 proto kernel scope link src 192.168.121.50 metric 100


--------------------------------------------------------------------------------
11.7 KEY LESSONS LEARNED FROM REAL SESSION
--------------------------------------------------------------------------------

LESSON 1: Always apply after modify
  nmcli con modify ... does NOT apply immediately
  Always run: nmcli connection up ens160

LESSON 2: NM stored routes vs kernel routes can differ
  ip r                                      = kernel routing table (live)
  nmcli connection show ens160 | grep route = what NM has saved (persistent)
  Always check NM stored routes before removing

LESSON 3: DHCP keeps coming back
  If DHCP IP returns after reboot, set method to manual:
  nmcli connection modify ens160 ipv4.method manual

LESSON 4: connection down/up clears leftover kernel routes
  nmcli connection down ens160
  nmcli connection up ens160

LESSON 5: Subnet mask matters
  192.168.121.55/21 is WRONG  (wrong subnet)
  192.168.121.55/24 is CORRECT
  Always verify with: ip a show ens160

LESSON 6: Kernel auto-route cannot be removed permanently
  192.168.121.0/24 dev ens160 proto kernel  = auto created by kernel
  It exists as long as an IP is assigned to the interface
  It is NORMAL and HARMLESS

LESSON 7: nmcli typos cause errors
  nmcli con mofify  = ERROR (typo)
  nmcli con modify  = CORRECT
  Always include connection name: nmcli con modify ens160 ...

--------------------------------------------------------------------------------
12.7 KEY LESSONS LEARNED - NTP
--------------------------------------------------------------------------------

LESSON 1: NTP needs DNS to resolve pool hostnames
  Always set DNS before configuring NTP with hostnames
  nmcli connection modify ens160 ipv4.dns "8.8.8.8 8.8.4.4"

LESSON 2: Empty resolv.conf = no DNS = NTP hostnames fail
  Check: cat /etc/resolv.conf
  Fix:   nmcli connection modify ens160 ipv4.dns "8.8.8.8 8.8.4.4"

LESSON 3: Gateway is NOT always an NTP server
  Just because gateway responds to ping does not mean it serves NTP

LESSON 4: Ping failure to NTP server does not mean NTP will fail
  NTP uses UDP port 123
  Ping uses ICMP
  Firewall may block ICMP but allow UDP 123
  Always test with chronyc sources, not ping

LESSON 5: Wait 60 seconds after chronyd restart
  Chrony needs time to poll sources and select best one
  systemctl restart chronyd && sleep 60 && chronyc sources -v

LESSON 6: Check logs for exact error
  journalctl -u chronyd -n 50
