1. SOLUTION OVERVIEW MAPPED TO FOCUS AREAS
--------------------------------------------------------------------
Focus Area 1 — Performance analysis
  Agent computes % change in IOPS, Latency, Throughput, Cache Hit
  Ratio between two snapshots and labels each as Improvement /
  Degradation / Neutral.

Focus Area 2 — Capacity reporting
  Agent computes Utilization %, Available Capacity, Used Capacity,
  and Growth Rate changes, and flags capacity risk.

Focus Area 3 — Configuration automation & drift detection
  Agent diffs configuration fields (RAID, Cache Policy, Replication,
  Tiering, Host Mappings, Pool Allocation, Dedup, Compression)
  against each other AND against a baseline standards table, then
  labels drift as Compliant / Minor / Critical.

All three feed one unified 8-section report:
  Executive Summary > Configuration Changes > Performance Impact >
  Capacity Impact > Configuration Drift > Risk Assessment >
  AI Insights > Recommended Actions

6. STEP 3 — FOCUS AREA 1: PERFORMANCE ANALYSIS (DUMMY WALKTHROUGH)
--------------------------------------------------------------------
Dummy Before (2026-06-01):
  IOPS = 12,000 | Latency = 8.5 ms | Throughput = 450 MB/s
  Cache Hit Ratio = 82% | Read/Write Ratio = 70/30

Dummy After (2026-07-01):
  IOPS = 14,160 | Latency = 7.5 ms | Throughput = 549 MB/s
  Cache Hit Ratio = 86% | Read/Write Ratio = 68/32

Calculation (formula: ((After - Before) / Before) x 100):
  IOPS:        ((14160 - 12000) / 12000) x 100 = +18.0%  -> Improvement
  Latency:     ((7.5 - 8.5) / 8.5) x 100 = -11.8%          -> Improvement (lower is better)
  Throughput:  ((549 - 450) / 450) x 100 = +22.0%          -> Improvement
  Cache Hit:   ((86 - 82) / 82) x 100 = +4.9%               -> Improvement

Expected agent output for this section:
  "Performance Impact:
   - IOPS increased by 18.0% (Improvement)
   - Latency reduced by 11.8% (Improvement)
   - Throughput increased by 22.0% (Improvement)
   - Cache hit ratio improved by 4.9% (Improvement)"

Validation step: Manually verify these 4 numbers match the formula
above before trusting the agent's live output — this is your
accuracy gate for the >99.5% success metric.


7. STEP 4 — FOCUS AREA 2: CAPACITY REPORTING (DUMMY WALKTHROUGH)
--------------------------------------------------------------------
Dummy Before (2026-06-01):
  Utilization = 78% | Used Capacity = 780 TB | Available = 220 TB
  Growth Rate = 3.5% / month

Dummy After (2026-07-01):
  Utilization = 65% | Used Capacity = 715 TB | Available = 385 TB
  Growth Rate = 1.2% / month

Calculation:
  Utilization change: 78% -> 65% = -13 percentage points
  Available capacity change: 385 - 220 = +165 TB increase
  Used capacity change: 715 - 780 = -65 TB decrease
  Growth rate change: 3.5% -> 1.2% = slowed significantly

Expected agent output:
  "Capacity Impact:
   - Utilization reduced from 78% to 65%
   - Available capacity increased by 165TB
   - Used capacity decreased by 65TB
   - Growth rate slowed from 3.5%/month to 1.2%/month"

Risk check: Utilization is under the 90% escalation threshold, so
no capacity-driven escalation should fire for this dummy record.


8. STEP 5 — FOCUS AREA 3: CONFIGURATION AUTOMATION & DRIFT
   DETECTION (DUMMY WALKTHROUGH)
--------------------------------------------------------------------
Dummy Before (2026-06-01) Configuration:
  RAID Policy = RAID5
  Cache Policy = Dynamic
  Replication = Async
  Tiering Policy = Enabled
  Host Mappings = 12 hosts
  Pool Allocation = Pool-01
  Deduplication = Enabled
  Compression = Enabled

Dummy After (2026-07-01) Configuration:
  RAID Policy = RAID6
  Cache Policy = Static
  Replication = Sync
  Tiering Policy = Disabled
  Host Mappings = 12 hosts (unchanged)
  Pool Allocation = Pool-01 (unchanged)
  Deduplication = Enabled (unchanged)
  Compression = Enabled (unchanged)

Dummy Baseline Standard (BaselineStandards table for this device
type):
  Standard RAID Policy = RAID6
  Standard Cache Policy = Dynamic
  Standard Replication = Sync
  Standard Tiering Policy = Enabled

Step-by-step diff logic:
  1. Compare Before vs After -> changed fields = RAID Policy,
     Cache Policy, Replication, Tiering Policy.
  2. Compare After vs Baseline:
     - RAID Policy: RAID6 vs RAID6 -> Compliant
     - Cache Policy: Static vs Dynamic -> Drift (Minor, since it
       does not remove redundancy or safety, but deviates from
       standard)
     - Replication: Sync vs Sync -> Compliant
     - Tiering Policy: Disabled vs Enabled -> Drift (Critical,
       since it deviates from a required standard and affects
       long-term cost/performance)

Expected agent output:
  "Configuration Changes:
   - RAID Policy changed from RAID5 to RAID6
   - Cache Policy changed from Dynamic to Static
   - Replication changed from Async to Sync
   - Tiering Policy changed from Enabled to Disabled

   Configuration Drift:
   - RAID Policy: Compliant with baseline
   - Replication: Compliant with baseline
   - Cache Policy: Minor drift (baseline = Dynamic)
   - Tiering Policy: Critical drift (baseline = Enabled, tiering
     disabled)"

Risk Assessment for this dummy record:
  - No redundancy reduced (RAID improved, not reduced)
  - Replication improved (Async -> Sync), so resilience is better
  - But Tiering disabled = Critical drift
  Overall Risk = Medium (per the example logic in the base system
  prompt: config drift without redundancy loss = Medium, not High)

12. MAPPING RESULTS TO SUCCESS METRICS
--------------------------------------------------------------------
  Faster insights:
    Time how long the dummy end-to-end run takes (upload -> report)
    vs a manual spreadsheet comparison of the same dummy numbers.
    Target: agent report in under 60 seconds vs manual analysis
    that typically takes 30-60+ minutes.

  Higher accuracy (>99.5%):
    Compare every numeric value in the agent's output against your
    manual calculations in Steps 3-4. Accuracy = (correct
    values / total values) x 100. All dummy-data test cases should
    hit 100% before moving to real data; monitor ongoing accuracy
    with periodic manual spot-checks in production.

  Reduced manual effort:
    Track number of manual steps eliminated: manual CSV comparison,
    manual percentage-change math, manual baseline lookup, manual
    Teams/email escalation drafting — all four are now automated.

  Scalable cloud-based solution:
    Confirm the same DEV agent, flows, and Dataverse tables can
    handle multiple devices (add Device_B, Device_C dummy rows) and
    multiple concurrent comparisons without manual reconfiguration.
    Add 2-3 more dummy devices and re-run Step 11's test script for
    each to prove scalability before go-live.
