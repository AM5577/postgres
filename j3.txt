------ COPY START (Full System Prompt for Instructions Field) ------
You are Storage Change Impact Intelligence Copilot, an enterprise AI assistant specialized in storage infrastructure analysis.

Your mission is to help storage administrators compare storage device configurations across two timelines, identify configuration changes, analyze performance and capacity impact, detect configuration drift, assess risk, and generate actionable recommendations.

BUSINESS PROBLEM:
Storage administrators frequently make configuration changes to storage arrays, clusters, pools, RAID policies, caching policies, replication settings, and host mappings. Understanding the impact of these changes is difficult and requires manual analysis across multiple systems. Key questions users want answered: What exactly changed? Which configuration parameters changed? Did performance improve or degrade? What capacity impact occurred? Is there configuration drift from standards? What is the likely business risk?

AGENT ROLE:
You are an intelligent storage operations assistant. Your responsibilities: Compare storage snapshots between two selected timelines. Detect configuration differences. Analyze performance changes. Analyze capacity changes. Detect configuration drift. Generate risk assessment. Provide executive summary. Recommend next actions. You must provide concise, accurate, and actionable responses.

INPUTS:
The user provides: Device Name, Device Type (Storage Array / Cluster / Pool), Before Change Snapshot, After Change Snapshot. Snapshots may include configuration parameters, capacity metrics, and performance metrics.

DATA FIELDS:
Configuration Data: RAID Policy, Cache Policy, Replication Settings, Tiering Policy, Host Mappings, Pool Allocation, Deduplication Settings, Compression Settings.
Performance Data: IOPS, Latency, Throughput, Cache Hit Ratio, Read/Write Ratio.
Capacity Data: Utilization %, Available Capacity, Used Capacity, Growth Rate.

ANALYSIS LOGIC:
Step 1 — Compare Configuration: Identify all configuration changes between before and after snapshots.
Step 2 — Analyze Performance Impact: Calculate percentage change using ((After - Before) / Before) x 100. Determine Improvement, Degradation, or Neutral impact. Higher IOPS = Positive. Lower Latency = Positive. Higher Throughput = Positive.
Step 3 — Analyze Capacity Impact: Evaluate utilization changes, capacity growth, and available capacity changes.
Step 4 — Detect Configuration Drift: Compare against enterprise baseline standards. Flag as Standard compliant, Minor drift, or Critical drift.
Step 5 — Risk Assessment: Assign severity as Low, Medium, High, or Critical based on risk factors including high utilization, performance degradation, unsafe configuration changes, disabled redundancy, and reduced replication.

OUTPUT FORMAT:
Always structure responses as follows: 1. Executive Summary 2. Configuration Changes 3. Performance Impact 4. Capacity Impact 5. Configuration Drift 6. Risk Assessment 7. AI Insights 8. Recommended Actions

ESCALATION RULES:
Escalate immediately if: Risk is High or Critical, Utilization exceeds 90%, Latency increases more than 30%, Redundancy is reduced, or Replication is disabled.

AUTOMATION:
Integrate with Power Automate for snapshot ingestion, scheduled comparisons, alert notifications, Teams notifications, email reports, and Power BI dashboard refresh.

SUCCESS METRICS:
Measure reduction in manual analysis time, faster root cause identification, increased storage efficiency, reduced configuration drift, and improved incident response time.
------ COPY END ------

8. BUILD STEP 6 — BUILD POWER AUTOMATE FLOWS
--------------------------------------------------------------------
Create these 4 flows in https://make.powerautomate.com (same
environment as your Dataverse tables):

FLOW A "Snapshot Ingestion"
  1. Trigger: "When a file is created" (SharePoint/OneDrive) or
     manual trigger from Copilot Studio file upload.
  2. Action: Parse CSV rows.
  3. Action: Validate required columns are present.
  4. Action: Dataverse "Add a new row" into StorageSnapshots for
     each parsed row.
  5. Action: Respond to Copilot Studio with success/failure.

FLOW B "GetSnapshotComparison"
  1. Trigger: "When Copilot triggers a flow" (Power Virtual
     Agents/Copilot Studio trigger), inputs = DeviceName,
     BeforeDate, AfterDate.
  2. Action: Dataverse "List rows" from StorageSnapshots filtered
     by DeviceName + BeforeDate -> store as BeforeRow.
  3. Action: Dataverse "List rows" filtered by DeviceName +
     AfterDate -> store as AfterRow.
  4. Action: Compose — calculate percentage changes using formula
     ((After - Before) / Before) x 100 for IOPS, Latency,
     Throughput, CacheHitRatio.
  5. Action: Compose — compare each configuration field between
     BeforeRow and AfterRow; build a list of changed fields.
  6. Action: Dataverse "List rows" from BaselineStandards filtered
     by DeviceType.
  7. Action: Compose — compare AfterRow config fields to baseline;
     assign Compliant/Minor/Critical per field.
  8. Action: Compose — apply risk scoring rules (see Section 9
     below).
  9. Action: Dataverse "Add a new row" into ComparisonResults with
     all computed values.
  10. Action: Respond to Copilot Studio with the full JSON payload
      matching the 8-section format.

FLOW C "EscalationAlert" — see Section 9 below.

FLOW D "PowerBIRefresh"
  1. Trigger: Recurrence (e.g., nightly) OR "When a flow completes"
     (chained after Flow B).
  2. Action: Power BI "Refresh a dataset".
  3. Action: Check refresh status; log failure to Dataverse if
     needed.

12. FOCUS AREA WALKTHROUGH 1: PERFORMANCE ANALYSIS (DUMMY MATH)
--------------------------------------------------------------------
Device_A Before (2026-06-01): IOPS 12,000 | Latency 8.5ms |
Throughput 450MB/s | Cache Hit 82%
Device_A After (2026-07-01): IOPS 14,160 | Latency 7.5ms |
Throughput 549MB/s | Cache Hit 86%

Formula: ((After - Before) / Before) x 100
  IOPS:       +18.0% -> Improvement
  Latency:    -11.8% -> Improvement (lower is better)
  Throughput: +22.0% -> Improvement
  Cache Hit:  +4.9%  -> Improvement

Expected output:
"Performance Impact:
 - IOPS increased by 18.0% (Improvement)
 - Latency reduced by 11.8% (Improvement)
 - Throughput increased by 22.0% (Improvement)
 - Cache hit ratio improved by 4.9% (Improvement)"

14. FOCUS AREA WALKTHROUGH 3: CONFIG DRIFT DETECTION (DUMMY MATH)
--------------------------------------------------------------------
Device_A Before config: RAID5, Cache=Dynamic, Replication=Async,
Tiering=Enabled
Device_A After config: RAID6, Cache=Static, Replication=Sync,
Tiering=Disabled
Baseline for Storage Array: RAID6, Cache=Dynamic, Replication=Sync,
Tiering=Enabled

Diff logic:
  1. Before vs After changes: RAID Policy, Cache Policy,
     Replication, Tiering Policy (4 changes).
  2. After vs Baseline:
     RAID Policy RAID6 = RAID6 -> Compliant
     Replication Sync = Sync -> Compliant
     Cache Policy Static vs Dynamic -> Minor drift
     Tiering Disabled vs Enabled -> Critical drift

Expected output:
"Configuration Changes:
 - RAID Policy changed from RAID5 to RAID6
 - Cache Policy changed from Dynamic to Static
 - Replication changed from Async to Sync
 - Tiering Policy changed from Enabled to Disabled

Configuration Drift:
 - RAID Policy: Compliant
 - Replication: Compliant
 - Cache Policy: Minor drift (baseline = Dynamic)
 - Tiering Policy: Critical drift (baseline = Enabled)"

Overall Risk = Medium (no redundancy lost, replication improved,
but one critical drift flag).

19. TROUBLESHOOTING
--------------------------------------------------------------------
Agent doesn't detect configuration changes correctly:
  Check CSV column headers match Flow B's expected field names
  exactly; confirm Flow A inserted rows correctly into Dataverse.

Escalation not firing on high risk:
  Check Flow B's RiskLevel output string matches Flow C's condition
  exactly (case-sensitive); re-test with Device_B/Device_C dummy
  rows.

Power BI dashboard shows stale data:
  Check Flow D's schedule/trigger and Power BI dataset refresh
  history; confirm Dataverse connector credentials are valid.

Agent gives inconsistent report formatting:
  Confirm the Output Format section of the system prompt (Section 4
  above) hasn't been edited; confirm generative orchestration isn't
  overriding the locked topic flow.

Users get "access denied":
  Check Entra ID group membership, Dataverse security role
  assignment, and DLP policy connector restrictions.
